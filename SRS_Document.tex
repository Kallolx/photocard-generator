\section{Introduction}

\subsection{Purpose}

The purpose of this Software Requirements Specification (SRS) document is to provide a comprehensive description of the Photocard News Card Generator Platform. This document outlines the functional and non-functional requirements, system architecture, user interfaces, and design constraints for the platform.

The Photocard platform is designed to serve journalists, content creators, media organizations, and social media managers who need to create professional, shareable news cards from article URLs or custom content. The system automates the extraction of article metadata, provides customization options, and enables high-quality image generation suitable for social media distribution.

This SRS is intended for:
\begin{itemize}
    \item Software developers and engineers implementing the system
    \item Quality assurance teams testing the platform
    \item Project managers overseeing development
    \item Stakeholders evaluating project progress
    \item System administrators deploying and maintaining the platform
\end{itemize}

\subsection{Scope}

The Photocard platform is a comprehensive web-based SaaS (Software as a Service) application that enables users to generate professional news cards from article URLs or custom content. The system consists of a Next.js frontend application and a Node.js/Express backend API with MySQL database.

\textbf{Key Capabilities:}
\begin{itemize}
    \item \textbf{URL-Based Card Generation:} Automatically extract article metadata (title, image, site name, favicon) from any news article URL and generate professional news cards
    \item \textbf{Custom Card Creation:} Create cards with custom logos, images, titles, and social media information
    \item \textbf{Multi-Language Support:} Interface available in English and Bengali languages
    \item \textbf{Tiered Subscription Plans:} Free, Basic, and Premium plans with varying feature access and usage limits
    \item \textbf{Batch Processing:} Generate multiple cards simultaneously (Premium feature)
    \item \textbf{Extensive Customization:} Customize backgrounds, gradients, patterns, themes, borders, fonts, and ad banners
    \item \textbf{API Access:} RESTful API for programmatic card generation (Premium feature)
    \item \textbf{Admin Dashboard:} Comprehensive user management, statistics, and system monitoring
    \item \textbf{Credit System:} Daily usage limits with automatic reset mechanism
    \item \textbf{Authentication \& Authorization:} Secure JWT-based authentication with role-based access control
\end{itemize}

\textbf{Out of Scope:}
\begin{itemize}
    \item Video content generation
    \item Real-time collaborative editing
    \item Mobile native applications (iOS/Android)
    \item Direct social media posting integration
    \item Payment gateway integration (currently manual via WhatsApp)
\end{itemize}

\section{General Description}

\subsection{Product Perspective}

The Photocard News Card Generator is a standalone web-based SaaS platform designed to address the needs of modern journalists and content creators who require quick, professional visual content for social media distribution.

\subsubsection{System Context}

The system operates as an independent web application with the following architectural components:

\textbf{Frontend Application:}
\begin{itemize}
    \item Next.js 16.1.4 with React 19.2.3
    \item Server-Side Rendering (SSR) and Client-Side Rendering (CSR)
    \item Responsive design using Tailwind CSS 4
    \item TypeScript for type safety
    \item Context API for state management (Auth, Language)
\end{itemize}

\textbf{Backend API:}
\begin{itemize}
    \item Node.js with Express.js framework
    \item RESTful API architecture
    \item MySQL 2 database with connection pooling
    \item JWT-based authentication
    \item Puppeteer for server-side rendering
    \item Rate limiting and security middleware
\end{itemize}

\textbf{External Integrations:}
\begin{itemize}
    \item Web scraping for URL metadata extraction
    \item Image proxy service for external image handling
    \item WhatsApp for manual payment coordination
    \item Email providers (Gmail, Yahoo, Outlook, etc.) for user registration
\end{itemize}

\textbf{System Boundaries:}
\begin{itemize}
    \item The system does not directly integrate with social media APIs
    \item Payment processing is manual via WhatsApp
    \item No third-party authentication providers (Google, Facebook login)
    \item No email service integration for notifications
\end{itemize}

\subsection{Functional Requirements}

\subsubsection{User Authentication \& Authorization}

\textbf{FR-1.1: User Registration}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall allow new users to register with name, email, and password
    \item \textbf{Input:} Name (2-255 chars), email (valid format, allowed domains), password (minimum 8 chars)
    \item \textbf{Processing:} Validate email domain, check for existing email, hash password, create user record, initialize credits (5/day for Free plan)
    \item \textbf{Output:} Success message with access token and refresh token, or error message
    \item \textbf{Validation:} Email must be from allowed domains (Gmail, Yahoo, Outlook, etc.)
\end{itemize}

\textbf{FR-1.2: User Login}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall authenticate users with email and password
    \item \textbf{Input:} Email and password
    \item \textbf{Processing:} Verify email exists, compare password hash, generate JWT tokens, update last login
    \item \textbf{Output:} User data with credits information and authentication tokens
    \item \textbf{Security:} Failed login attempts logged, account status checked
\end{itemize}

\textbf{FR-1.3: Token-Based Authentication}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall use JWT tokens for session management
    \item \textbf{Access Token:} 7-day expiration (configurable)
    \item \textbf{Refresh Token:} 30-day expiration (configurable)
    \item \textbf{Storage:} localStorage and HTTP cookies
    \item \textbf{Validation:} Token verified on each protected request
\end{itemize}

\textbf{FR-1.4: Role-Based Access Control}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall enforce different access levels based on user role
    \item \textbf{Roles:} User (default), Admin
    \item \textbf{Admin Access:} User management, statistics, plan modifications, system settings
    \item \textbf{User Access:} Card generation, profile management, API key management (Premium)
\end{itemize}

\textbf{FR-1.5: Profile Management}
\begin{itemize}
    \item \textbf{Priority:} Medium
    \item \textbf{Description:} Users shall be able to update name, email, and password
    \item \textbf{Constraints:} Email must be unique, password requires current password verification
    \item \textbf{Validation:} Same as registration requirements
\end{itemize}

\subsubsection{Card Generation Features}

\textbf{FR-2.1: URL Metadata Extraction}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall automatically extract article metadata from provided URLs
    \item \textbf{Extraction Sources:} Schema.org JSON-LD, Open Graph tags, Twitter Cards, HTML meta tags, H1 tags
    \item \textbf{Data Extracted:} Title, main image, site logo, favicon, site name, URL
    \item \textbf{Fallback:} If extraction fails, use default values or prompt user
    \item \textbf{Processing:} HTML entity decoding, string cleanup, domain-specific handling
\end{itemize}

\textbf{FR-2.2: Single Card Generation}
\begin{itemize}
    \item \textbf{Priority:} High
    \item \textbf{Description:} System shall generate individual news cards from URLs or custom input
    \item \textbf{Input:} Article URL or custom data (logo, image, title)
    \item \textbf{Processing:} Extract metadata, apply theme and customizations, render card
    \item \textbf{Output:} High-resolution PNG image (1200x630 recommended)
    \item \textbf{Credit Deduction:} 1 credit per card
\end{itemize}

\textbf{FR-2.3: Batch Card Generation}
\begin{itemize}
    \item \textbf{Priority:} Medium
    \item \textbf{Description:} Premium users shall be able to generate multiple cards simultaneously
    \item \textbf{Access:} Premium plan only
    \item \textbf{Input:} Multiple URLs (array)
    \item \textbf{Processing:} Process each URL in parallel, track individual status
    \item \textbf{Output:} Array of cards with individual status (completed, error)
    \item \textbf{Batch Download:} ZIP file containing all generated cards
    \item \textbf{Credit Deduction:} 1 credit per card in batch
\end{itemize}

\textbf{FR-2.4: Custom Card Creation}
\begin{itemize}

\subsection{Use Cases}

\subsubsection{Use Case \#1: Generate News Card from URL}

\textbf{Use Case ID:} UC-1

\textbf{Actor:} Registered User (Journalist)

\textbf{Preconditions:}
\begin{itemize}
    \item User is authenticated
    \item User has available daily credits
    \item User has valid article URL
\end{itemize}

\textbf{Main Success Scenario:}
\begin{enumerate}
    \item User navigates to URL generation page
    \item User selects "Single" mode
    \item User pastes article URL into input field
    \item User clicks "Generate" button
    \item System validates URL format
    \item System checks user's remaining credits
    \item System fetches article webpage
    \item System extracts metadata (title, image, logo, site name)
    \item System applies default theme and background
    \item System renders card preview
    \item System deducts 1 credit from user's daily allowance
    \item System displays success message
    \item User views card preview
    \item User customizes background color (optional)
    \item User clicks "Download" button
    \item System generates high-resolution PNG image
    \item Browser downloads card image
\end{enumerate}

\textbf{Alternative Flows:}
\begin{itemize}
    \item \textbf{A1: Invalid URL Format}
        \begin{enumerate}
            \item At step 5, system detects invalid URL format
            \item System displays error message
            \item Use case returns to step 3
        \end{enumerate}
    \item \textbf{A2: Insufficient Credits}
        \begin{enumerate}
            \item At step 6, system detects zero remaining credits
            \item System displays limit reached message
            \item System shows upgrade modal
            \item Use case ends
        \end{enumerate}
    \item \textbf{A3: Metadata Extraction Failure}
        \begin{enumerate}
            \item At step 8, system cannot extract metadata
            \item System uses fallback values or prompts user for manual input
            \item Use case continues from step 9
        \end{enumerate}
\end{itemize}

\textbf{Postconditions:}
\begin{itemize}
    \item Card is generated and displayed
    \item User credit is deducted
    \item Card generation is recorded in database
    \item User has downloaded card image
\end{itemize}

\subsubsection{Use Case \#2: Batch Generate Multiple Cards (Premium)}

\textbf{Use Case ID:} UC-2

\textbf{Actor:} Premium User (Social Media Manager)

\textbf{Preconditions:}
\begin{itemize}
    \item User has Premium plan subscription
    \item User is authenticated
    \item User has sufficient daily credits or unlimited plan
    \item User has multiple article URLs

\textbf{Main Success Scenario:}
\begin{enumerate}
    \item User navigates to URL generation page
    \item User clicks "Batch" mode button
    \item User enters first article URL in input field
    \item User clicks "Add URL" button
    \item Additional URL input fields appear
    \item User enters 2-5 more article URLs
    \item User clicks "Generate All" button
    \item System validates all URL formats
    \item System checks user's Premium plan status
    \item System checks available credits
    \item System processes each URL in parallel
    \item For each URL, system extracts metadata
    \item System renders card preview for each URL
    \item System updates individual card status (loading → completed)
    \item System deducts credits (1 per card)
    \item System displays all generated cards
    \item User reviews all cards
    \item User clicks "Download All" button
    \item System creates ZIP file containing all cards
    \item Browser downloads ZIP file
\end{enumerate}

\textbf{Alternative Flows:}
\begin{itemize}
    \item \textbf{A1: Non-Premium User Attempts Batch}
        \begin{enumerate}
            \item At step 2, system detects user plan is not Premium
            \item System displays upgrade modal
            \item Use case ends
        \end{enumerate}
    \item \textbf{A2: Some URLs Fail}
        \begin{enumerate}
            \item At step 12, some URLs fail metadata extraction
            \item System marks those cards with error status
            \item System continues processing remaining URLs
            \item System generates cards for successful URLs only
            \item System displays error summary
            \item Use case continues from step 16
        \end{enumerate}
    \item \textbf{A3: Insufficient Credits for Batch}
        \begin{enumerate}
            \item At step 10, system detects insufficient credits for all cards
            \item System displays error message indicating how many cards can be generated
            \item System offers to generate partial batch or cancel
            \item Use case ends or continues with partial batch
        \end{enumerate}
\end{itemize}

\textbf{Postconditions:}
\begin{itemize}
    \item Multiple cards are generated
    \item Credits deducted (1 per successful card)
    \item All generations recorded in database
    \item ZIP file downloaded with all cards
\end{itemize}

\subsection{Non-Functional Requirements}

\subsubsection{Performance}

\textbf{NFR-1.1: Response Time}
\begin{itemize}
    \item \textbf{Requirement:} 95\% of API requests shall respond within 2 seconds under normal load
    \item \textbf{Page Load:} Initial page load shall complete within 3 seconds on broadband connection
    \item \textbf{Card Generation:} Single card shall generate within 5 seconds
    \item \textbf{Batch Processing:} Batch of 5 cards shall complete within 15 seconds
    \item \textbf{Measurement:} Response time measured from request initiation to complete response
\end{itemize}

\textbf{NFR-1.2: Throughput}
\begin{itemize}
    \item \textbf{Requirement:} System shall support at least 100 concurrent users
    \item \textbf{API Throughput:} Handle minimum 50 API requests per minute
    \item \textbf{Database:} Support minimum 500 queries per minute
    \item \textbf{Scalability:} Architecture shall allow horizontal scaling
\end{itemize}

\textbf{NFR-1.3: Resource Utilization}
\begin{itemize}
    \item \textbf{Memory:} Backend process shall not exceed 2GB RAM under normal load
    \item \textbf{Database:} Connection pool limited to 10 concurrent connections
    \item \textbf{Browser:} Client application shall not exceed 500MB browser memory
    \item \textbf{CPU:} Puppeteer rendering shall not block other requests
\end{itemize}

\textbf{NFR-1.4: Image Quality}
\begin{itemize}
    \item \textbf{Resolution:} Generated cards shall be high DPI (2x device pixel ratio)
    \item \textbf{Format:} PNG with transparency support
    \item \textbf{File Size:} Optimized to under 2MB per card
    \item \textbf{Quality:} No visible artifacts or compression issues
\end{itemize}

\subsubsection{Reliability}

\textbf{NFR-2.1: Availability}
\begin{itemize}
    \item \textbf{Uptime:} System shall maintain 99\% uptime during business hours
    \item \textbf{Downtime:} Planned maintenance windows announced 24 hours in advance
    \item \textbf{Recovery:} System shall recover from failures within 15 minutes
\end{itemize}

\textbf{NFR-2.2: Error Handling}
\begin{itemize}
    \item \textbf{Graceful Degradation:} System shall continue operating when non-critical components fail
    \item \textbf{User Feedback:} All errors shall be presented with clear, actionable messages
    \item \textbf{Logging:} All errors shall be logged with timestamp, user context, and stack trace
    \item \textbf{Recovery:} Failed card generations shall not corrupt user credit balance
\end{itemize}

\textbf{NFR-2.3: Data Integrity}
\begin{itemize}
    \item \textbf{Transactions:} All database operations shall use transactions where appropriate
    \item \textbf{Consistency:} Credit deductions shall be atomic with card generation
    \item \textbf{Validation:} Input validation on both client and server side
    \item \textbf{Backups:} Database backups performed daily
\end{itemize}

\textbf{NFR-2.4: Fault Tolerance}
\begin{itemize}
    \item \textbf{Database:} Automatic reconnection on connection loss
    \item \textbf{External Services:} Fallback mechanisms for failed metadata extraction
    \item \textbf{API:} Retry logic for transient failures
\end{itemize}

\subsubsection{Availability}

\textbf{NFR-3.1: Service Availability}
\begin{itemize}
    \item \textbf{Target:} 99\% availability during peak hours (9 AM - 9 PM local time)
    \item \textbf{Maintenance:} Maximum 4 hours planned downtime per month

\section{Change Management Process}

\subsection{Process Overview}

The change management process ensures that all changes to requirements are evaluated, approved, and documented systematically. This process maintains project stability while allowing necessary adaptations.

\subsection{Change Request Procedure}

\subsubsection{Step 1: Change Identification}
\begin{itemize}
    \item Stakeholder identifies need for requirement change
    \item Change can originate from: users, developers, testers, management, or external factors
    \item Initial impact assessment performed
\end{itemize}

\subsubsection{Step 2: Change Request Submission}
\begin{itemize}
    \item Requester submits formal Change Request (CR) document
    \item CR includes:
    \begin{itemize}
        \item CR ID (unique identifier)
        \item Date submitted
        \item Requester name and role
        \item Affected requirement ID(s)
        \item Current requirement description
        \item Proposed change description
        \item Justification/rationale
        \item Priority (High, Medium, Low)
        \item Estimated impact (schedule, cost, resources)
    \end{itemize}
\end{itemize}

\subsubsection{Step 3: Change Review}
\begin{itemize}
    \item Change Control Board (CCB) reviews CR
    \item CCB composition: Project Manager, Lead Developer, QA Lead, Stakeholder Representative
    \item Review criteria:
    \begin{itemize}
        \item Business value and justification
        \item Technical feasibility
        \item Impact on existing functionality
        \item Resource requirements
        \item Schedule implications
        \item Risk assessment
    \end{itemize}
\end{itemize}

\subsubsection{Step 4: Decision}
\begin{itemize}
    \item CCB makes decision: Approve, Reject, or Defer
    \item Decision recorded with rationale
    \item If approved: assign priority and schedule implementation
    \item If rejected: document reasons and communicate to requester
    \item If deferred: specify conditions for future consideration
\end{itemize}

\subsubsection{Step 5: Implementation}
\begin{itemize}
    \item Development team implements approved changes
    \item Update SRS document with change details
    \item Update version number and revision history
    \item Notify all stakeholders of changes
\end{itemize}

\subsubsection{Step 6: Verification}
\begin{itemize}
    \item QA team verifies implementation
    \item Regression testing performed
    \item Stakeholder acceptance obtained
    \item Close change request
\end{itemize}

\subsection{Change Classification}

\textbf{Minor Changes:}
\begin{itemize}
    \item Clarifications or corrections to existing requirements
    \item UI text changes
    \item Non-functional improvements
    \item Approval: Lead Developer + Project Manager
\end{itemize}

\textbf{Major Changes:}
\begin{itemize}
    \item New features or functionality
    \item Changes to core business logic
